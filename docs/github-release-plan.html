ChatMap GitHub 发布计划

本计划用于把 ChatMap 从本地开发项目整理成可在 GitHub 上公开发布、可被他人理解、构建、试用和反馈的开源项目。 当前建议以 v0.1.0 pre-release 发布，定位为 Edge / ChatGPT 扩展的早期可用版本。

发布目标

让用户能通过 README 理解项目、从源码构建、在 Edge 中加载未打包扩展，并通过 GitHub Release 下载预览包。

发布范围

当前对话思维导图、节点跳转、布局、编辑、AI 摘要与链接、导入导出、Side Panel / Full Page / Float。

发布策略

建议先 GitHub 公开源码和预览包，再准备 Edge Add-ons。商店版负责普通用户安装与自动更新，GitHub 负责透明开发和预览测试。

核心风险

ChatGPT 页面结构变化、AI 兼容接口差异、重复问题定位不准、扩展商店图标与权限审核要求。

Product Readiness

目标：让主界面清爽，降低首次用户理解成本

Settings Page

  • 新增独立设置页，入口放在扩展 action、View 菜单或右键菜单中。
  • 迁移全局设置：AI Provider、Base URL、Model、API Key、Auto summarize、Default layout、Float 默认状态。

  • 设置页加入版本信息、Release notes 链接、Update status 区域。

ChatGPT Floating Launcher

  • 当用户打开 ChatGPT 对话页时，在屏幕右侧显示 ChatMap 悬浮球。
  • 悬浮球位置避开 ChatGPT 原有悬浮按钮，并允许用户关闭或移动。
  • 左键点击悬浮球：自动打开 ChatMap Side Panel。
  • 右键点击悬浮球：打开 ChatMap Settings Page。
  • 悬浮球状态本地保存，避免用户关闭后反复打扰。
这两个功能建议作为 GitHub 首发前的产品整理项。它们不会改变核心图谱能力，但会显著减少主界面按钮密度，并让用户在 ChatGPT 页面中更自然地发现 ChatMap。

阶段 8：发布策略与 GitHub Release

Store vs GitHub Strategy

目标：选择首发路径，避免更新体验和审核节奏混乱
策略 优点 缺点 适合 ChatMap 吗
GitHub-first 最快发布；源码透明；方便收集开发者反馈；可快速迭代 pre-release。 普通用户需要开发者模式加载；无法真正一键自动更新；信任门槛较高。 适合作为当前 0.1.0 早期预览。
Store-first 普通用户安装最顺；浏览器负责自动更新；信任度高；权限和隐私材料更规范。 审核有延迟；上架素材要求更多；早期频繁改动会被审核节奏拖慢。 适合核心体验稳定后作为公开主渠道。
Hybrid GitHub 保持源码、issue、预览包；商店负责稳定版安装与更新；大型开源插件常见这种形态。 需要维护两个发布节奏：GitHub release 可能早于商店审核通过。 推荐。先 GitHub pre-release，再准备 Edge Add-ons。
推荐路径：先发布 GitHub v0.1.0 pre-release，明确这是开发者模式安装包；随后准备 Edge Add-ons 素材、隐私材料、PNG icon 和审核说明。稳定后以商店版作为普通用户主入口，GitHub 继续作为源码、问题反馈和预览版渠道。

阶段 9：发布后维护与更新

Post-Release Operations

目标：把反馈转化为路线图

Update Delivery

  • 商店安装版：由 Edge Add-ons 或 Chrome Web Store 自动更新，用户通常不需要手动操作。
  • GitHub / unpacked 安装版：ChatMap 只能提示有新版本，不能安全地替用户静默替换扩展文件。
  • 实现 Update Notice：插件打开时检查最新 release/version manifest，显示版本、摘要和风险等级。
  • 用户自主选择：Open Release Page、Download ZIP、Remind Later、Ignore This Version。
  • 设置页保留更新设置：检查频率、是否显示 pre-release、最近忽略版本。

One-Click Update Feasibility

对普通公开用户，真正的一键自动更新应通过插件商店完成。开发者模式加载的 dist 或 GitHub zip 不能由扩展自己直接覆盖安装，这是浏览器扩展安全模型的一部分。

Edge 支持带 update_url 的外部安装扩展更新机制，但这更适合企业或自托管 CRX 场景，不适合作为普通用户首发主路径。
