TurnMap 0.7.0: Knowledge Organization And Node Editing

Version goal: improve node-level knowledge organization while preserving source jumping, saved graph recovery, user edits, manual layout, and existing topic/link workflows.

Non-Negotiable Boundaries

  • Do not overwrite user-authored titles, summaries, links, tags, colors, collapsed state, important state, source anchors, or custom notes.
  • Do not break ChatGPT jump stability, sourceAnchor/sourceAnchors recovery, multi-site adapters, themes, language packs, or API task logs.
  • New UI copy must ship in English and Chinese and remain compatible with generated custom language packs. This includes dialogs, confirmations, inline notices, option labels, placeholders, status-bar messages, task-log entries, disabled-state hints, and error text. Avoid browser-native prompt text for new workflows because it bypasses TurnMap's 0.5.x AI custom translation surface.
  • New file-related actions should use the existing Files menu. New node/link operations should use existing Node Actions or Link Actions surfaces unless there is a concrete reason not to.

Decision Log To Fill

1. Node Expansion MVP

Decision: 0.7.0 will use a structured mini-map / outline inside the node, not a full nested React Flow editor. The MVP should store durable expansion data and support readable rendering plus constrained edits such as title/summary edits, color, fold, importance, and deletion. Full internal manual linking and complete nested graph parity are deferred to a later 0.7.x or 0.8+ iteration after the data model and interaction model prove stable.

2. Node Expansion Storage Boundary

Decision: expanding a node must not delete or overwrite the node's original AI-answer summary. Store the decomposed result in a separate node-owned structured field such as expandedContent / answerMap. Normal display mode keeps reading the original title and summary; expanded display mode reads the structured answer map. Restoring normal display only switches the display mode back and keeps the expansion data unless the user explicitly deletes the expansion result. Decision: node expansion is API-only for 0.7.0. Do not implement local rule-based decomposition fallback in this release. If no API key/provider is configured, the expand action should be disabled or guide the user to AI settings through localized UI copy. If the AI request fails, returns empty output, or returns a structure that does not pass validation, do not write any expansion data; keep the node exactly as it was, and report the failure in the status bar and task log. AI expansion output should create compact mini nodes with title only; do not store or render mini-node summaries in 0.7.0. The minimum mini-node count depends on the source answer's actual structure, but a useful expansion should normally contain at least two meaningful mini nodes unless the source answer is genuinely too short. The prompt should prefer 6-18 mini nodes and enforce a hard maximum of 24. Results beyond the hard maximum should be rejected or reduced to higher-level structure before write-in. Mini-map links should favor section, subpoint, and summary relationships within the same local structure, while lowering the weight of cross-branch links so the embedded view stays clean and avoids excessive line crossings. Mini-map editing scope for 0.7.0 is deliberately constrained. Required: edit mini-node title, color mini nodes, mark mini nodes important, delete mini nodes, delete related internal links when a mini node is removed, and save/restore these edits. Deferred: mini-node summary fields, mini-node subtree fold/unfold, manual mini-node creation, manual internal linking, internal link type/color/importance editing, and freeform mini-node dragging. The outer TurnMap node still keeps its existing fold/unfold behavior. If deleting mini nodes leaves fewer than two mini nodes, keep the expanded state but show localized guidance that the expansion is no longer structurally useful and the user can restore the normal answer display or run AI expansion again. Do not automatically switch display modes as a hidden side effect of deletion. Node expansion controls should live in the existing Node Actions panel as single-node actions. For one selected ordinary AI turn node, show Expand Answer when no expansion exists. For an expanded node, show localized controls for Show Original, Show Expanded, Delete Expansion, and Re-expand as appropriate. Do not support batch expansion in 0.7.0, because it would create many API tasks and many write opportunities at once. Show Original and Delete Expansion must be separate actions. Show Original only switches display mode back to the normal AI answer while preserving expansion data. Show Expanded switches back to the mini-map view. Delete Expansion requires confirmation and removes expansion data only after confirmation. Re-expand also requires confirmation because it may overwrite the existing AI expansion and any user edits made inside the mini-map. Export boundary: TurnMap JSON must fully export expansion data and the current display mode so it can restore the complete editable map. PNG export must render the currently visible expanded mini-map state. At least one editable mind-map export format should preserve the expansion structure as much as the format allows; candidate formats should be evaluated from the existing export set, likely Obsidian Canvas first because it is already editable and graph-oriented. Documentation must clearly tell users to prefer TurnMap JSON when they want complete save/restore fidelity for expanded nodes and other TurnMap-specific editing state. Obsidian Canvas is the required editable graph export target for expanded nodes in 0.7.0. When exporting an expanded TurnMap node, keep the outer node as a parent-level Canvas node preserving the TurnMap node title/original answer semantics, and export mini nodes as nearby editable Canvas nodes with internal links representing the expansion structure. Do not flatten the mini-map into a static image inside Canvas. Obsidian Canvas import is not required to round-trip expanded nodes back into TurnMap expansion data in 0.7.0. Canvas is an editable external export/work format, not the complete TurnMap persistence format. Full fidelity save/restore is guaranteed only through TurnMap JSON. AI expansion input should prefer the original assistant answer/source text for that turn, not the user-edited title or summary. If the original answer text is unavailable, fall back to the current summary and record that fallback in the task log. Expansion writes must still preserve the user-edited outer node title and summary.

3. Resize Precedence

Decision: size/layout precedence is Rebuild > user manual resize > refresh / deep scan recovery > automatic layout. Rebuild may intentionally regenerate dimensions and layout. Once the user manually resizes a node, ordinary refresh, deep scan, conversation switching, and automatic layout must preserve that user size. Automatic layout may still choose initial positions and default sizes for nodes that have no user size. Expanded mini-map auto-sizing may enlarge a node only when no user manual size exists. If refresh or deep scan remaps node IDs, stored dimensions should migrate with the original node when the remap confidently identifies the same source turn. If identity is uncertain, use default dimensions rather than applying the wrong size to another answer. Show Original / Show Expanded may switch between normal default size and expanded auto-size only when the node has no user manual size. Once a user manually resizes a node, display-mode changes must preserve the user size. Resize interaction should expose persistent drag zones on the left edge, right edge, bottom edge, lower-left corner, and lower-right corner. Do not expose top-edge or upper-corner resize zones in 0.7.0, so the node header and drag handle area remain less conflicted with node movement and selection. User-facing documentation must explain where the resize drag zones are located. Resized nodes must have minimum width and height limits so users cannot save a node state where the title, tags, core controls, or expanded mini-map become effectively unusable. Resized nodes should also have maximum width and height protection, tuned for Side Panel and Full Page use, so a saved node cannot permanently cover most of the canvas or become hard to recover. The maximum should still leave enough room for expanded mini-map readability.

4. Topic Collapse Reversibility

Decision: topic collapse may group non-contiguous selected nodes. A topic group is semantic, not necessarily a continuous turn range. The implementation must persist the grouped node IDs, original positions, and enough edge/layout metadata to restore the original nodes, original connections, and layout when expanded. While collapsed, external links that touched grouped nodes should be represented by system-generated proxy links between the external node and the topic node so the map remains connected. Proxy links must be marked as generated collapse-view edges and must not become permanent user links. Expanding the topic removes proxy links and restores the original links. The topic node is an editable long-lived group record rather than a disposable placeholder. User edits to its title, summary, tags, color, importance, and display state should persist as topic-group metadata. Expanding the topic may hide the topic node from the current view, but it should not delete the group record unless the user explicitly deletes the topic group. 0.7.0 will not add hidden group-management UI for collapsing back to an expanded topic. After a topic is expanded, the group record remains in data. If the user later selects the same member set and collapses it again, reuse the existing topic metadata instead of creating a fresh duplicate group. Nested topic collapse is out of scope and should be blocked in 0.7.0. A topic node cannot be selected with other nodes to create a larger topic group, and members that already belong to an active collapsed topic should not be collapsed into another topic.

5. Batch Scope

Decision: node tag management should be integrated into the existing Node Actions panel. For selected nodes, the panel should support localized add-tag and remove-tag controls instead of relying on browser prompt dialogs. It should show the selected nodes' tag union for removal; removing a tag removes it from every selected node that has it and skips nodes that do not. All labels, placeholders, buttons, status messages, and errors must use English and Chinese i18n keys so built-in language switching and AI-generated custom language packs can cover them. Tag normalization should be light: trim surrounding whitespace, enforce a maximum length of 32 characters, and de-duplicate tags within each node. Preserve user casing and non-English text exactly after trimming; do not force lowercase or translate user-authored tag content. Batch link operations should be triggered by selecting multiple links, not by selecting multiple nodes. When multiple links are selected, Link Actions enters a batch mode for type, color, and importance edits. Multi-selecting nodes should only affect node operations unless the user explicitly selects links. Link color semantics should remain type-driven for 0.7.0. Batch relationship-type changes apply that type's default color. Batch importance changes affect emphasis only. Do not introduce independent custom link-color overrides unless the existing link model already has a clear field for it. Logging boundary: AI or potentially slow tasks such as node expansion, re-expansion, provider failures, and structure validation failures must report both status-bar messages and task-log entries. Fast local edits such as batch tags, batch link changes, resize save, and topic collapse/expand should use localized status-bar feedback without filling the API task log.

Open Risks

Largest risk: "small mind map inside a node supports existing edit capabilities" can balloon into a second graph editor inside every node. That may conflict with React Flow drag, resize, selection, link creation, and source-jump gestures.
Persistence risk: topic collapse and node expansion both want to hide or replace visible content while preserving original source-backed state. Without a clear schema, refresh/rebuild/import may silently lose user work.
Conservative path: start with durable dimensions, then store expansion data as node-owned structured content, then render it as a constrained mini-map before attempting full nested graph editing parity.

Implementation Order

  1. Extend persistence schema and test fixtures first so dimensions, expansion data, topic groups, and batch edits share one save/load path.
  2. Implement node resizing, dimension precedence, remap migration, min/max constraints, and documentation for resize drag zones.
  3. Implement API-only node expansion data model, AI request, validation, atomic write behavior, and Show Original / Show Expanded display switching.
  4. Implement compact mini-map rendering, constrained mini-node editing, and TurnMap JSON / PNG / Obsidian Canvas export support.
  5. Replace prompt-based tag operations with localized Node Actions tag add/remove controls.
  6. Implement true multi-link selection batch edits for relationship type and importance with type-driven colors.
  7. Implement reversible topic group records, proxy links while collapsed, metadata reuse for repeated same-member collapse, and no nested topic collapse.
  8. Finish README, Chinese README, changelog, user guide, i18n completeness tests, unit tests, typecheck, build, and package a local testable unpacked release after verification passes.
